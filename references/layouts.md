# 레이아웃 가이드

livedeck는 슬라이드 레이아웃 12종을 지원한다. 어떤 레이아웃을 언제 쓰고, 각 레이아웃이 어떤 필드를 받아 화면에서 어떻게 동작하는지 안내한다. 문서 후반에는 입력을 레이아웃으로 자동 매핑하는 규칙도 함께 다룬다.

## 1. 개요

슬라이드 레이아웃은 `slides.json`의 각 슬라이드 객체에서 `"layout"` 필드로 지정한다. 지원하는 값은 다음 12종이다.

| 레이아웃 | 한 줄 용도 |
|----------|-----------|
| `COVER` | 표지(첫 슬라이드) |
| `SECTION` | 챕터/구간 구분 |
| `CONTENT` | 본문 내용(불릿) |
| `IMAGE` | 이미지 + 텍스트(또는 이미지 그리드) |
| `STAT` | 큰 수치 카드 그리드(성능·지표 강조) |
| `CARDS` | 아이콘+제목+본문 카드 그리드(기능·항목 나열) |
| `TABLE` | 표(스펙·비교·매핑) |
| `CALLOUT` | 강조 박스(주의·팁·핵심 메시지, 시맨틱 컬러) |
| `COMPARE` | 좌우 2단 비교(전/후, A/B, 장/단) |
| `TIMELINE` | 가로 단계/절차(번호 노드 + 연결선 + 카드) |
| `CODE` | 코드/CLI 블록(어두운 배경, 줄별 렌더) |
| `CLOSING` | 마무리/감사/CTA |

`layout`을 지정하지 않거나 위 12종에 없는 값을 넣으면 `CONTENT`로 폴백된다. 즉 어떤 입력이 와도 빈 슬라이드가 되지는 않지만, 의도한 화면을 얻으려면 열두 값 중 하나를 정확히 적어야 한다.

### 인라인 강조 서식 (모든 텍스트 필드 공통)

`title` / `subtitle` / `body` 항목 / 카드·표 셀 등 청중 화면에 보이는 텍스트에는 다음 인라인 서식을 쓸 수 있다.

| 서식 | 입력 | 화면 |
|------|------|------|
| 굵게 | `**핵심**` | 진하게(ink) |
| 하이라이트 | `==25만==` | 노란 형광펜 |
| 코드 | `` `mount -t s3files` `` | 모노스페이스 코드 |

수치, 핵심 키워드, 명령어/식별자에 적극적으로 써서 시각적 위계를 만든다.

### 시맨틱 컬러 (tone)

`STAT`·`CARDS`·`CALLOUT`·`COMPARE` 의 항목에는 `tone` 으로 색을 줄 수 있다. 값: `accent`(기본·중립강조) / `info`(파랑·참고) / `success`(초록·장점) / `warn`(주황·주의) / `danger`(빨강·금지·미지원). 미지정 시 `accent`(CALLOUT 은 `info`).

레이아웃에 따라 footer 표시 여부도 갈린다. `COVER`와 `CLOSING`은 footer를 자동으로 숨기고, `SECTION`과 `CONTENT`는 (footer 설정이 켜져 있으면) footer를 표시한다.

## 2. 레이아웃별 상세

### COVER (표지)

- 언제 쓰는가: 덱의 첫 슬라이드. 발표 제목과 한 줄 설명을 보여주는 표지.
- 받는 필드
  - `section`: 제목 위에 작게 올라가는 eyebrow 문구(선택). 발표 시리즈명이나 분류 라벨에 쓴다.
  - `title`: 표지 제목. 큰 글씨로 렌더된다.
  - `subtitle`: 제목 아래 한 줄 설명(선택).
- 화면 동작: 내용이 세로 중앙에 오고, 좌측 정렬된다. footer는 자동으로 숨겨진다.
- 좋은 예: `section`에 "Sample Deck", `title`에 발표 주제, `subtitle`에 그 주제를 한 문장으로 푼 설명.
- 피할 예: 표지에 불릿 목록을 넣으려는 시도. COVER는 `body`를 렌더하지 않으므로 목록은 사라진다.

```
+--------------------------------------+
|                                      |
|  EYEBROW (선택)                      |
|  큰 제목 (display)                    |
|  부제 (subtitle)                      |
|                                      |
+--------------------------------------+   footer 없음
```

### SECTION (구간 구분)

- 언제 쓰는가: 챕터가 바뀌거나 큰 주제로 전환할 때. 청중에게 "여기서 이야기가 넘어간다"를 알리는 칸막이.
- 받는 필드
  - `title`: 구간 제목. 이 레이아웃은 제목 하나만 받는다.
- 화면 동작: 가는 헤어라인(rule)이 제목 위에 그어지고, 큰 타이틀이 세로 중앙에 놓인다. footer는 표시된다.
- 좋은 예: "어떻게 동작하나"처럼 다음 묶음을 한마디로 예고하는 제목.
- 피할 예: 부제나 본문을 함께 기대하는 것. SECTION은 `subtitle`도 `body`도 렌더하지 않는다. 설명이 필요하면 바로 뒤에 CONTENT 슬라이드를 둔다.

```
+--------------------------------------+
|  ---- (짧은 룰)                       |
|  섹션 제목 (display)                   |
+--------------------------------------+
|  (c) ...        n/total        logo   |   footer 표시
```

### CONTENT (본문)

- 언제 쓰는가: 설명, 목록, 근거 등 실제 내용을 전달하는 대부분의 슬라이드. 덱의 본문 대다수가 여기에 해당한다.
- 받는 필드
  - `title`: 슬라이드 제목.
  - `body`: 불릿으로 표시할 항목 배열. 각 원소가 하나의 불릿이 된다.
- 화면 동작: 제목이 상단에 고정되고, 불릿 목록은 남은 공간에서 세로 중앙에 정렬된다. footer는 표시된다.
- 좋은 예: 제목 한 줄과, 한눈에 들어오는 분량의 짧은 불릿 몇 개.
- 피할 예: 한 슬라이드에 불릿을 화면이 넘칠 만큼 몰아넣는 것. 한 불릿에 긴 문장을 통째로 넣는 것.

```
+--------------------------------------+
|  제목 (h2)                            |
|   - 불릿 1                            |
|   - 불릿 2                            |
|   - 불릿 3                            |
+--------------------------------------+
|  (c) ...        n/total        logo   |   footer 표시
```

### IMAGE (이미지 + 텍스트)

- 언제 쓰는가: 스크린샷, 다이어그램, 사진 등 이미지를 보여줄 때. 이미지 한 장과 설명을 나란히 두거나, 여러 장을 격자로 펼친다.
- 받는 필드
  - `title`: 슬라이드 제목(모든 variant 공통). `<h2>`로 상단에 놓인다.
  - `variant`: `left` | `right` | `full` 중 하나. 미지정 시 기본 `right`. 그 외 값은 `right`로 폴백한다.
  - `body`: 불릿 배열. `left`/`right`에서 텍스트 영역에 표시된다. `full`에서는 렌더하지 않는다.
  - `images`: 이미지 객체 배열. 각 객체는 `src`(필수)와 `caption`(선택)을 가진다. 캡션이 있으면 이미지 아래 얇은 룰과 함께 작은 muted 텍스트로 붙는다.
- 화면 동작(variant별)
  - `left`: 2분할. 왼쪽 이미지, 오른쪽 텍스트(title + body).
  - `right`(기본): 2분할. 왼쪽 텍스트(title + body), 오른쪽 이미지.
  - `full`: 2분할 없는 풀블리드. 제목은 상단에 두고, 이미지가 본문 영역을 채운다. body는 렌더하지 않는다.
  - 이미지가 2장 이상이면 variant와 무관하게 이미지 영역이 `repeat(auto-fit, minmax(220px, 1fr))` 격자로 자동 배치된다. 수량에 종속되지 않는다.
- footer는 다른 본문 레이아웃과 동일하게 표시된다(자동 숨김 없음).
- 좋은 예: `right`에 스크린샷 한 장 + 그 화면을 설명하는 짧은 불릿. 또는 `full`에 다이어그램 한 장 + 한 줄 캡션.
- 피할 예: `full`에 불릿을 기대하는 것(렌더되지 않는다). 한 슬라이드에 이미지를 격자가 빽빽해질 만큼 몰아넣는 것. 캡션에 긴 문단을 넣는 것.

```
right (기본): 텍스트 | 이미지        left: 이미지 | 텍스트
+--------------------------------+   +--------------------------------+
|  제목 (h2)                      |   |  제목 (h2)                      |
|  - 불릿 1     |   [ 이미지 ]     |   |  [ 이미지 ]    |  - 불릿 1       |
|  - 불릿 2     |   caption       |   |  caption      |  - 불릿 2       |
+--------------------------------+   +--------------------------------+
|  (c) ...   n/total   logo       |   |  (c) ...   n/total   logo       |

full: 풀블리드 (body 렌더 안 함)      images 2장 이상: auto-fit 격자
+--------------------------------+   +--------------------------------+
|  제목 (h2)                      |   |  제목 (h2)                      |
|       [   이미지   ]            |   |  [ img ] [ img ] [ img ]        |
|       caption                  |   |  cap     cap     cap            |
+--------------------------------+   +--------------------------------+
|  (c) ...   n/total   logo       |   |  (c) ...   n/total   logo       |
```

### STAT (통계 카드)

- 언제 쓰는가: 성능 수치, 지표, 가격 같은 "숫자"를 강조할 때. 표의 한 줄로 묻히던 수치를 큰 글씨 카드로 띄운다.
- 받는 필드
  - `title`: 슬라이드 제목.
  - `items`: 통계 카드 배열. 각 항목은 `value`(큰 수치, 필수), `label`(수치 설명), `sub`(부연·단위, 선택), `tone`(선택).
- 화면 동작: `auto-fit` 카드 그리드. 각 카드는 상단에 tone 색 5px 보더, 큰 컬러 수치, 라벨, 부연 순. 카드 3~6개가 보기 좋다.
- 좋은 예: "최대 읽기 IOPS" → value `250,000`, label "읽기 IOPS", sub "파일시스템당".
- 피할 예: value 에 긴 문장. value 는 숫자나 짧은 토큰만.

```
+--------------------------------------+
|  제목 (h2)                            |
|  ┌────────┐ ┌────────┐ ┌────────┐    |
|  │ 250K   │ │ ~1ms   │ │ 25,000 │    |  ← 큰 컬러 수치
|  │ IOPS   │ │ 지연   │ │ 동시연결│    |
|  └────────┘ └────────┘ └────────┘    |
+--------------------------------------+
```

### CARDS (카드 그리드)

- 언제 쓰는가: 기능, 항목, 단계 등 동급 요소 여러 개를 나열할 때. 평평한 불릿 대신 카드로 나눠 한눈에 구조가 보이게 한다.
- 받는 필드
  - `title`: 슬라이드 제목.
  - `items`: 카드 배열. 각 항목은 `title`(카드 제목), `icon`(이모지 한 글자, 선택), `body`(설명 문자열 또는 불릿 배열, 선택), `tone`(선택).
- 화면 동작: `auto-fit` 카드 그리드. 카드는 좌측 tone 색 보더 + 아이콘 배지 + 제목 + 본문. 카드 2~6개 권장.
- 좋은 예: 보안 항목 4개를 각각 아이콘+제목+한 줄 카드로.
- 피할 예: 카드 안에 본문을 문단으로 길게. 카드 본문은 짧은 구·불릿으로.

```
+--------------------------------------+
|  제목 (h2)                            |
|  ┌─────────┐ ┌─────────┐             |
|  │ 🔒 제목  │ │ ⚡ 제목  │             |
|  │ 본문...  │ │ 본문... │             |
|  └─────────┘ └─────────┘             |
+--------------------------------------+
```

### TABLE (표)

- 언제 쓰는가: 스펙 비교, 항목-값 매핑, 서비스 대조처럼 행/열 구조가 본질인 데이터. 원본 마크다운에 표가 있으면 표로 그대로 살린다.
- 받는 필드
  - `title`: 슬라이드 제목.
  - `headers`: 헤더 셀 배열(선택). 있으면 강조색 헤더 행.
  - `rows`: 행 배열. 각 행은 셀 배열. **각 행의 첫 셀은 자동으로 강조(th, accent-soft 배경)** 된다.
- 화면 동작: 둥근 모서리 + 그림자 표. 헤더는 accent 배경/흰 글씨, 첫 열은 옅은 강조, 짝수 행은 옅은 줄무늬.
- **행 수 자동 맞춤**: 빌드 시 (헤더+본문) 행 수가 `data-rows` 로 부여되고, 7행 이상부터 폰트·패딩이 단계적으로 축소되어 제목 아래 영역에 들어간다(최대 ~15행). 즉 행이 많아도 제목을 가리거나 넘치지 않는다. 다만 행이 많을수록 글씨가 작아지므로 **가독성 한계(약 12행)** 를 넘으면 표를 둘로 나누거나 핵심 행만 추린다.
- 좋은 예: 4열 이하, 6행 이하면 기본 크기로 시원하게. 셀은 짧게.
- 피할 예: 셀에 긴 문장. 열이 5개를 넘어 가로로 글씨가 깨질 때 → 열을 줄이거나 CARDS 로 전환. (세로 행 수는 자동 축소되지만 가로 열 폭은 자동 축소되지 않는다.)

```
+--------------------------------------+
|  제목 (h2)                            |
|  ┃ 헤더1  │ 헤더2  │ 헤더3 ┃  ← accent |
|  ┃ 행키   │ 값     │ 값    │           |
|  ┃ 행키   │ 값     │ 값    │           |
+--------------------------------------+
```

### CALLOUT (강조 박스)

- 언제 쓰는가: 한 슬라이드에서 가장 중요한 메시지, 주의/경고, 핵심 팁을 크게 띄울 때. 시맨틱 컬러로 성격(참고/장점/주의/금지)을 색으로 전달한다.
- 받는 필드
  - `title`: 슬라이드 제목.
  - `tone`: `info`(기본) / `success` / `warn` / `danger` / `accent`.
  - `icon`: 이모지 한 글자(선택). 예: "⚠️", "💡", "✅".
  - `heading`: 박스 안 굵은 한 줄(선택).
  - `body`: 본문 문자열 또는 불릿 배열.
- 화면 동작: tone-soft 배경 + 좌측 tone 색 6px 보더 박스. 아이콘 + heading + 본문. 화면 세로 중앙.
- 좋은 예: "AL2023 기본 efs-utils 2.4.2 는 S3 Files 미지원" → tone `warn`, icon "⚠️".
- 피할 예: 콜아웃 박스를 한 슬라이드에 여러 개. 강조의 핵심은 하나로 모으는 것.

```
+--------------------------------------+
|  제목 (h2)                            |
|  ┃⚠️ heading (굵게)                   |  ← warn-soft 배경
|  ┃   본문 / 불릿                       |     좌측 warn 보더
+--------------------------------------+
```

### COMPARE (좌우 비교)

- 언제 쓰는가: 전/후, 기존 방식 vs 새 방식, A안 vs B안, 장점 vs 단점처럼 두 묶음을 나란히 대조할 때.
- 받는 필드
  - `title`: 슬라이드 제목.
  - `left` / `right`: 각각 `{ heading, tone?, body:[불릿] }`.
- 화면 동작: 1:1 두 칼럼 카드. 각 칼럼은 상단 tone 색 보더 + heading(밑줄) + 불릿. 보통 left 는 중립/부정(`accent`·`warn`), right 는 긍정(`success`)으로 대비를 준다.
- 좋은 예: 기존 ML 파이프라인(왼쪽, warn) vs S3 Files 사용(오른쪽, success).
- 피할 예: 3개 이상 비교(→ CARDS). 한 칼럼에 불릿을 화면 넘치게.

```
+--------------------------------------+
|  제목 (h2)                            |
|  ┌──────────┐  ┌──────────┐          |
|  │ 기존 방식 │  │ S3 Files  │          |
|  │ - ...     │  │ - ...     │          |
|  └──────────┘  └──────────┘          |
+--------------------------------------+
```

### TIMELINE (가로 단계/절차)

- 언제 쓰는가: 순서가 있는 단계, 절차, 로드맵, 프로세스. 번호가 매겨진 흐름을 가로로 보여준다.
- 받는 필드
  - `title`: 슬라이드 제목.
  - `steps`: 단계 배열. 각 항목은 `title`(단계 제목), `label`(STEP 1 같은 라벨, 선택), `body`(설명 문자열 또는 불릿 배열, 선택), `tone`(선택).
- 화면 동작: 단계가 가로로 균등 배치되고, 각 단계 위에 번호 노드(원형) + 단계 사이 연결선. 노드 아래 세로 카드(라벨/제목/본문). 단계 3~5개 권장.
- 좋은 예: 설치 절차 4단계, 각 단계 제목 + 한 줄 설명.
- 피할 예: 6단계 이상(카드가 좁아짐 → 둘로 나눔). 단계 본문을 길게.

```
+--------------------------------------+
|  제목 (h2)                            |
|  ①────②────③────④                   |
|  ┌──┐ ┌──┐ ┌──┐ ┌──┐                 |
|  │카드│ │카드│ │카드│ │카드│            |
|  └──┘ └──┘ └──┘ └──┘                 |
+--------------------------------------+
```

### CODE (코드/CLI 블록)

- 언제 쓰는가: 명령어, 코드 스니펫, 터미널 출력. 어두운 배경에 모노스페이스로 보여준다.
- 받는 필드
  - `title`: 슬라이드 제목.
  - `intro`: 코드 위 한 줄 설명(선택).
  - `lines`: 코드 줄 배열(권장. 들여쓰기·기호 그대로 보존) **또는** `code`: 여러 줄 문자열(`\n` 으로 분리).
- 화면 동작: 어두운 코드 블록(`#1e1e2e`) + 밝은 모노 텍스트. 줄별로 렌더해 공백/들여쓰기가 보존된다. **코드 안에서는 인라인 서식(`**`/`==`/`` ` ``)이 적용되지 않는다**(그대로 출력).
- 좋은 예: 명령 2~10줄. 프롬프트(`$`)와 출력을 함께.
- 피할 예: 화면을 넘치는 긴 코드(핵심만 발췌). 한 줄이 너무 길어 줄바꿈되는 경우(짧게 끊기).

```
+--------------------------------------+
|  제목 (h2)                            |
|  intro 설명 (선택)                    |
|  ┌────────────────────────────┐      |
|  │ $ command --flag           │      |  ← 어두운 배경
|  │ output line                │      |
|  └────────────────────────────┘      |
+--------------------------------------+
```

### CLOSING (마무리)

- 언제 쓰는가: 발표의 마지막. 감사 인사, 연락처, 다음 행동 요청(CTA).
- 받는 필드
  - `title`: 마무리 제목.
  - `subtitle`: 연락처나 CTA 한 줄(선택).
- 화면 동작: 배경이 어둡게 반전되고 글씨가 흰색으로 나온다. 내용은 세로 중앙, 좌측 정렬된다. footer는 자동으로 숨겨진다.
- 좋은 예: `title`에 "고맙습니다", `subtitle`에 이메일 주소나 다음 단계 안내.
- 피할 예: 마무리에 본문 목록을 넣는 것. CLOSING은 `body`를 렌더하지 않는다.

```
+--------------------------------------+   반전 surface (어두움)
|  큰 제목 (흰 글자)                     |
|  CTA / 연락처 (subtitle)              |
+--------------------------------------+   footer 없음
```

## 3. 상황별 권장 흐름

레이아웃을 고를 때의 기본 대응은 다음과 같다.

- 표지 -> `COVER`
- 목차나 구간 전환 -> `SECTION`
- 설명/목록 -> `CONTENT`
- 감사/연락처/CTA -> `CLOSING`

일반적인 덱은 대략 이런 순서로 구성된다.

1. `COVER` 표지로 연다.
2. (선택) `SECTION`으로 첫 묶음을 예고한다.
3. `CONTENT` 슬라이드로 내용을 풀어 간다. 주제가 바뀔 때마다 `SECTION`을 한 장씩 끼워 칸막이를 둔다.
4. `CLOSING`으로 닫는다.

`SECTION`은 필요할 때만 넣는다. 짧은 덱이라면 COVER -> CONTENT 여러 장 -> CLOSING만으로도 충분하다.

## 4. 안티패턴

- CONTENT의 불릿이 화면을 넘칠 만큼 많으면 슬라이드를 나눈다. 기준은 고정 개수가 아니라 "한눈에 들어오는 분량"이다. 청중이 한 화면을 훑어 바로 이해할 수 있는지를 보고 판단한다.
- 한 불릿에 긴 문장을 그대로 넣지 않는다. 핵심을 구(句) 단위로 줄여 적는다. 풀어 말할 내용은 슬라이드가 아니라 발표자 대본(`script`)에 둔다.
- `COVER`와 `CLOSING`에 본문 목록을 욱여넣지 않는다. 이 두 레이아웃은 `body`를 렌더하지 않으므로, 넣어도 화면에 나오지 않는다.
- `SECTION`은 `title`만 받는다. 부제나 본문을 기대하지 않는다. 구간 설명이 필요하면 SECTION 다음에 CONTENT를 둔다.
- 지원하지 않는 `layout` 값을 적지 않는다. 12종 외의 값은 모두 `CONTENT`로 폴백되므로, 의도한 화면을 얻지 못한다.

## 5. 실제 샘플 해설

`examples/sample_slides.json`의 5장으로 각 레이아웃이 실제로 어떻게 쓰이는지 본다.

1. "livedeck 소개" (`COVER`): 덱의 첫 슬라이드이자 표지다. `section`("Sample Deck"), `title`, `subtitle`로 발표 주제와 한 줄 설명을 보여주므로 COVER가 맞다.
2. "무엇을 해결하나" (`CONTENT`): 제목 아래에 해결 항목 세 가지를 불릿(`body`)으로 나열하는 본문 슬라이드라 CONTENT다.
3. "어떻게 동작하나" (`SECTION`): 동작 원리 설명으로 넘어가기 직전의 구간 칸막이다. 제목 하나로 주제 전환을 알리므로 SECTION이다.
4. "동기화 구조" (`CONTENT`): 앞 SECTION이 예고한 동작 원리를 불릿 세 개로 풀어내는 본문이라 CONTENT다.
5. "고맙습니다" (`CLOSING`): 마지막 슬라이드. `title`("고맙습니다")과 `subtitle`(연락처 이메일)로 마무리하므로 CLOSING이다.

## 6. 자동 매핑 규칙

마크다운 등 입력을 레이아웃으로 자동 변환할 때(인제스천)의 기본 대응 규칙이다. 사용자가 슬라이드별 `layout`을 명시하면 그 값이 항상 우선한다.

1. 첫 슬라이드는 항상 `COVER`.
2. 마지막 슬라이드가 마무리/감사/CTA 성격이면 `CLOSING`.
3. 마크다운 입력에서 H1 또는 큰 주제 전환은 `SECTION`.
4. **마크다운 표(`| ... |`)는 `TABLE`로 그대로 옮긴다.** 평평한 불릿으로 뭉개지 않는다. 단 열이 너무 많아 표로 안 들어가면 핵심 열만 남기거나 `CARDS`로 바꾼다.
5. **수치·지표가 핵심인 묶음(성능 스펙, 가격, KPI)은 `STAT`.** 큰 숫자를 카드로 띄운다.
6. **동급 항목 3개 이상을 나열(기능 목록, 구성요소, 단계)** 하면 `CONTENT` 불릿 대신 `CARDS`를 우선 고려한다.
7. **주의/경고/금지/핵심 팁 한 덩어리는 `CALLOUT`** (tone: warn/danger/info/success). 원문의 ⚠️·"주의"·"필수"·"미지원" 신호가 단서.
8. **전/후, 기존 vs 신규, 장 vs 단처럼 두 묶음 대조는 `COMPARE`.**
9. **번호가 매겨진 순서·단계·절차·로드맵은 `TIMELINE`.** "1단계/2단계", "먼저/다음/마지막" 신호가 단서.
10. **명령어·코드·터미널 출력 블록은 `CODE`.** 마크다운 코드펜스(``` ```bash ```)나 `$` 프롬프트가 단서.
11. 이미지(`images`)가 포함된 슬라이드는 `IMAGE`. 텍스트가 함께 있으면 `right`(기본), 이미지만 있으면 `full`.
12. 그 외 일반 제목 + 불릿 구조는 `CONTENT`.
13. 사용자가 슬라이드별 `layout`을 명시하면 그 값이 항상 우선한다.

핵심 지침: **원문에 표/수치/대조/경고/절차/코드가 있으면 그 구조를 살리는 전용 레이아웃(TABLE/STAT/COMPARE/CALLOUT/TIMELINE/CODE)을 먼저 쓴다.** 모든 것을 CONTENT 불릿으로 평탄화하면 디자인이 빈약해진다. 텍스트 필드에는 인라인 서식(`**굵게**`/`==하이라이트==`/`` `코드` ``)을 적극 쓴다(단 CODE 본문은 예외 — 그대로 출력).
